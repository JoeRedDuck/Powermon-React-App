# Claude Workflow Instructions

## Testing and Deployment Process

Always follow this workflow before pushing changes to GitHub:

### 1. Run Tests
```bash
cd /home/joe/Documents/Prozomix/new_webserver
python3 -m pytest test_api.py test_db.py -v
```

**Requirements:**
- All tests must pass (37/37)
- Fix any failing tests before proceeding
- Warnings are acceptable (deprecation warnings for FastAPI on_event)

### 2. Check Git Status
```bash
git status
```

Review all modified and untracked files to ensure only intended changes are staged.

### 3. Stage Changes
```bash
git add <file1> <file2> ...
```

**Files to typically stage:**
- `app.py` - API endpoints
- `db.py` - Database functions
- `test_api.py` - API tests
- `test_db.py` - Database tests
- Any new `.md` documentation files

**Files to NEVER stage:**
- `__pycache__/` - Compiled Python files
- `.env` - Environment variables with secrets
- Database files or connection strings

### 4. Commit Changes
```bash
git commit -m "Brief description

- Bullet point of change 1
- Bullet point of change 2
- Bullet point of change 3"
```

**Good commit message example:**
```
Add monitor management API endpoints

- Add POST /api/v1/monitors/{id}/unassign endpoint
- Add DELETE /api/v1/monitors/{id} endpoint  
- Add POST /api/v1/monitors/{id}/reassign endpoint
- Add database functions for monitor operations
- Update tests to match new API response format
```

### 5. Push to GitHub
```bash
git push origin database_changes
```

Replace `database_changes` with the current branch name if different.

### 6. Verify Push Success
Check the terminal output for:
- "Writing objects: 100%" 
- Remote branch update confirmation
- No error messages

## Quick Command Reference

### Run all tests:
```bash
python3 -m pytest test_api.py test_db.py -v
```

### Run specific test file:
```bash
python3 -m pytest test_api.py -v
```

### Run with coverage:
```bash
python3 -m pytest test_api.py test_db.py --cov=. --cov-report=term-missing
```

### Check database state:
```bash
python3 -c "
from database import SessionLocal
import db
with SessionLocal() as session:
    monitors = db.get_monitors(session)
    for m in monitors:
        print(f'ID: {m[\"id\"]}, MAC: {m[\"mac\"]}, Assigned: {m[\"name\"]}')"
```

### Restore monitor assignments from settings.json:
```bash
python3 -c "
from database import SessionLocal
import db
with SessionLocal() as session:
    db.reassign_monitor(session, 1, 'Pump 1')
    db.reassign_monitor(session, 2, 'Pump 2')
    db.reassign_monitor(session, 3, 'Condenser 4')
    db.reassign_monitor(session, 4, 'Condenser 3')
    db.reassign_monitor(session, 5, 'Condenser 2')
    db.reassign_monitor(session, 6, 'Minus 80')
    db.reassign_monitor(session, 7, 'Condenser 1')"
```

## Important Notes

1. **Always run tests before pushing** - This catches breaking changes early
2. **Keep commits focused** - One logical change per commit
3. **Write descriptive commit messages** - Future you will thank you
4. **Don't commit sensitive data** - Check files before staging
5. **Test API changes with curl** - Manual testing complements automated tests
6. **Document new endpoints** - Update or create markdown documentation
7. **Check database state after tests** - Ensure test cleanup worked properly
